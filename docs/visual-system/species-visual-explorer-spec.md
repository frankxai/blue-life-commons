# Species Visual Explorer Spec

Last updated: 2026-07-03

## Purpose

Create a site surface where contributors can see which animal each image belongs to, which species still need media, and whether a visual is official, partner-granted, open-license, rich-embed-only, or generated supporting art.

## First Read

"Every species has a media provenance card: image, source, rights, review, and next action."

## Data Source

Primary data comes from:

- `content/species/**/*.md` for the science artifact
- `content/media/species-media-registry.yaml` for media status, candidates, and approved assets
- `content/media/species-media-source-routing.yaml` for source lanes, partner/grant targets, and reviewer actions
- `content/media/species-media-rich-embeds.yaml` for verified source-card fallbacks when approved images are absent
- `content/media/species-media-approval-queue.yaml` for curator decisions, checklist progress, and approved-primary templates
- `content/media/species-media-render-contract.yaml` for the public render strategy allowed for each species
- `content/media/species-media-curation-workspace.yaml` for batch order, blockers, and per-species next curator actions
- `content/media/species-media-site-data.json` for frontend-ready joined data
- `content/media/species-media-public-explorer-manifest.yaml` for the public-safe explorer read model with candidate URLs omitted and acquisition-plan metadata joined
- `content/media/public/species-visual-explorer-YYYY-MM-DD.html` as the current static public-safe prototype
- `content/media/review-packs/species-media-visual-board-YYYY-MM-DD.html` as the current static reviewer prototype
- `content/media/species-media-trace-ledger.yaml` for ownership/audit joins across page, source card, render contract, curation lane, and review-only candidate ID
- `content/media/species-media-acquisition-plan.yaml` for public-safe official/open-license/partner next actions without review-only candidate URLs
- `content/media/species-media-approval-workbench.yaml` for the reviewer-only joined approval surface with candidate preview URLs, Commons rights evidence, outreach state, trace ownership, missing checks, approval YAML starters, and promotion commands
- species-page `media` frontmatter generated by `scripts/sync_species_page_media.py --write`
- `VISUAL-ASSETS-REGISTRY.md` for generated/concept asset history

The site should never infer species ownership from filename alone. It must join by `artifact_id` and `species_page`.

Use the public explorer manifest or species-page `media` frontmatter for public pages and site prototypes. Use `species-media-site-data.json` and the review board only in reviewer tooling, because those records preserve candidate image fields for curator inspection.

Use the trace ledger for audit views, handoffs, and QA dashboards that need to prove which species owns which visual/source record. It should be safe to expose as metadata because it omits direct candidate image URLs and Commons file-page URLs.

Use the acquisition plan for public-safe next-action views, partner outreach planning, and source-priority dashboards. It should not replace the approval queue or curation workspace because it intentionally omits restricted candidate source links.

Use the approval workbench only for restricted reviewer tooling. It intentionally includes candidate image URLs and Commons file pages so curators can inspect the actual image, but it is marked `not_public_site_input` and must not feed public cards, species heroes, SEO images, social crops, or downloadable media.

Public explorer cards should expose the acquisition lane, target source family, partner/NGO outreach signal, and next public-safe curator action from the acquisition plan. They should not expose review-only candidate pages, direct image URLs, candidate titles, or thumbnails.

## Views

### Coverage Board

Dense grid of species with:

- primary media thumbnail or source-card placeholder
- common name and scientific name
- taxon group
- media status chip
- render strategy chip
- public fallback/source-card status chip
- curation decision/checklist status chip
- source family chip
- acquisition lane chip
- next action

### Media Inspector

Side panel or detail route with:

- approved asset or candidate preview
- source URL and original media URL
- source-card fallback URL and verifier status
- curator decision and checklist completion count
- render strategy, public visual kind, and whether hero image reuse is allowed
- acquisition lane, target source family, outreach target, and next public-safe action
- creator and credit line
- license and commercial-use note
- species-match basis
- ethics/location notes
- approved and blocked surfaces
- supporting generated assets, clearly separated from official media

### Curation Queue

Grouped worklist:

- `01_source_card_recheck`: verified source-card fallback is not yet public-safe
- `02_ethics_flag_review`: image candidate has welfare/proximity/crop flags
- `03_official_public_domain_fast_track`: likely official or institutional public-domain candidate
- `04_open_public_domain_review`: public-domain/CC0-style candidate without the same official signal
- `05_open_license_attribution`: CC BY / CC BY-SA candidate needing exact attribution and surface obligations
- `06_partner_grant_or_new_candidate`: reuse is unclear or a stronger partner/NGO image is needed

Each item should show promotion readiness, missing checks, blockers, source links, and the exact post-approval promotion commands.

## Interaction

Filters:

- taxon group
- media status
- rights status
- source-card fallback status
- render strategy
- curation decision
- source family
- acquisition lane
- target source family
- has supporting generated art
- has official/institutional candidate

Sorts:

- species page path
- status severity
- taxon group
- recently updated

Actions:

- copy attribution
- open source
- open species page
- open asset path
- mark candidate for review through a PR
- copy the approval queue record for curator review
- open the curation workspace batch for the species

## Rendering Rules

- Use approved primary media in species heroes.
- Use verified source cards when image rights are blocked or no primary image has been approved.
- Treat source-card public use as link/display permission only; it does not authorize image download, crops, social graphics, or hero image reuse.
- Use generated images only as supporting/editorial media and label them as generated concept plates.
- Never display unapproved media as the lead species identity image.
- Always show credit and license near the image or in an accessible media details panel.
- Public pages must render from `render_contract.public_visual`, not from `candidate.image_url`.
- Public explorer pages should prefer `species-media-public-explorer-manifest.yaml`, which removes candidate file pages, candidate direct image URLs, candidate titles, and review thumbnails entirely.
- Species detail pages may read their local `media` frontmatter block, but that block must be generated from the central contracts and must not contain candidate image URLs or Commons file-page links.
- `render_strategy: verified_source_card_fallback` can fill the page visual slot, but `species_page_hero_image_allowed` stays false until an approved primary image exists.
- `render_strategy: source_card_recheck_placeholder` and `review_only_candidate_placeholder` show non-public placeholders and reviewer actions, not photos.

## Accessibility And Ethics

- Every visible image needs alt text.
- Credit and license text must be keyboard reachable.
- Motion is unnecessary for the default board; use simple hover/focus states only.
- Do not expose precise sensitive locations through captions, EXIF-derived text, maps, or crop context.
