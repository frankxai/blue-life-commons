# Species Media Registry

This folder tracks animal and habitat-forming species visuals separately from prose artifacts.

The canonical queue is [`species-media-registry.yaml`](species-media-registry.yaml). Every file under `content/species/` must have one record there, even when the primary image is still needed. This prevents a species from silently shipping without an image, rich source embed, rights status, credit, or review trail.

Generated key art may support a page, campaign, or section mood, but it is never the official identification image. Primary species media must come from an official/institutional source, partner grant, owned contributor media, public-domain media, or open-license media with image-level attribution and ethics QA.

Public site surfaces should read `species-media-render-contract.yaml` or the joined `render_contract` object in `species-media-site-data.json`. They may render approved primary images, verified source-card fallbacks, or placeholders. The 2026-07-03 contract renders approved primary images for all 31 tracked species. Public routes must not render candidate image URLs as species identity media.

For a ready public read model, use `species-media-public-explorer-manifest.yaml`. It omits candidate image URLs, candidate file pages, candidate titles, and review-only thumbnail fields. It also joins public-safe acquisition lane, target source family, outreach target, and next-action metadata from `species-media-acquisition-plan.yaml`. The generated public prototype lives under `public/species-visual-explorer-YYYY-MM-DD.html` and renders only approved public visuals selected by the render contract.

Every species page should carry a compact `media` frontmatter block generated by `scripts/sync_species_page_media.py`. That block points to the registry, render contract, and public explorer record for the same `artifact_id`, exposes the public source-card embed URL, and mirrors approved primary image fields when approval exists.

The trace ledger, `species-media-trace-ledger.yaml`, is the ownership proof layer. It joins species page, registry, source-card fallback, render contract, public explorer record, approval/curation state, and review-only candidate ID without exposing candidate direct image URLs.

The acquisition plan, `species-media-acquisition-plan.yaml`, is the next-action layer. It says which species need official/public-domain review, open-license attribution review, ethics-first review, or partner/NGO media outreach, while keeping review-only candidate direct URLs out of public-safe metadata.

The approval dossiers, `species-media-approval-dossiers.yaml`, are reviewer-only evidence packets. They include candidate file pages and direct image URLs, so they must not feed public site surfaces. Use them to inspect the candidate, source card, acquisition lane, missing approval checks, missing approved-primary fields, and promotion commands for each species.

The Commons rights snapshots, `species-media-commons-rights-snapshots.yaml`, are reviewer-only source metadata captures from the Wikimedia Commons API. They compare the staged candidate against current Commons file metadata, license fields, creator/credit fields, species text signals, and automated ethics/location flags. They do not approve media or replace human rights, species-match, crop, alt-text, and ethics review.

The outreach packets, `species-media-outreach-packets.yaml`, are public-safe operating packets for source owners, partners, NGOs, institutions, and open-license publishers. They group species by target source domain, provide email-ready permission asks, list required license/credit/welfare fields, and keep candidate direct image URLs out of the packet. Outreach never approves media; it only helps gather the written permission or image-level terms that a curator later records in the approval queue.

The approval workbench, `species-media-approval-workbench.yaml`, joins the approval queue, approval dossiers, Commons rights snapshots, acquisition plan, outreach packets, public explorer, and trace ledger into one reviewer-only operating surface. It intentionally contains candidate file pages and direct image URLs, so it must not feed public site routes. Use the generated HTML workbench in `review-packs/` to inspect the candidate image, rights evidence, species text signal, public fallback, outreach target, missing checks, missing approved-primary fields, approval YAML starter, and promotion command for the same animal.

Curators should use `species-media-curation-workspace.yaml` or the matching review-pack Markdown to process candidates in batches: ethics-first candidates, official/public-domain fast track, open public-domain review, open-license attribution review, and partner-grant/replacement work.

The storage policy, `species-media-storage-policy.yaml`, defines the Vercel-Blob-first object-storage boundary: GitHub stores reviewable metadata, Vercel Blob holds the first approved image mirrors, Postgres later tracks operational state, and Vercel renders the application. The generated `species-media-storage-manifest.yaml` maps each approved species image to deterministic pathnames. The committed `species-media-blob-manifest.json` records the public Blob URLs that production routes prefer when an approved asset has been uploaded.
